# Landing API Service

Бэкенд-сервис для лендинг-презентации с интеграцией AI-инструментов.
Проект полностью контейнеризирован, поэтому для локального запуска не требуется предварительная настройка PHP или Composer.

## 1. Как запустить проект

Для запуска проекта вам понадобятся Docker и Docker Compose.

1. Клонируйте репозиторий.
2. Создайте файл окружения на основе примера: `cp .env.example .env`.
3. Если вы хотите протестировать работу AI, добавьте ваш ключ `OPENAI_API_KEY` в файл `.env`.
   *Обратите внимание: если ключ не указан или API недоступен, сработает механизм graceful fallback. Сервис продолжит работу и вернет шаблонный ответ, форма будет успешно обработана.*
4. Запустите контейнеры и выполните первоначальную настройку:
   ```bash
   docker-compose up -d --build
   docker-compose exec app composer install
   docker-compose exec app php artisan key:generate
   docker-compose exec app php artisan migrate
   docker-compose exec app php artisan l5-swagger:generate

5. API будет доступен по адресу: http://localhost:8080
Swagger-документация: http://localhost:8080/api/documentation

## Стек технологий
    Backend: PHP 8.4, Laravel 13
    База данных: MySQL 8.0
    Веб-сервер: Nginx (Alpine)
    Документация: OpenAPI 3.0 (пакет darkaonline/l5-swagger)
    AI: OpenAI API (GPT-3.5-turbo)

## Архитектура
Для сохранения чистоты кода и разделения ответственности в проекте использован слоистый подход:

Controllers: Отвечают только за прием HTTP-запроса, валидацию данных (через FormRequest) и возврат JSON-ответа.
Services: Содержат всю бизнес-логику. ContactService выступает в роли оркестратора, вызывая AiAnalysisService и EmailNotificationService.
Обоснование: Такое разделение упрощает поддержку кода и позволяет легко заменять реализации (например, сменить AI-провайдера или подключить реальный SMTP для отправки писем) без изменения логики контроллеров.

## Реализация API

POST /api/contact — отправка формы обратной связи. Реализована валидация входных данных и rate limiting (10 запросов в минуту). Возвращает статусы 200, 422, 429 и 500.
GET /api/health — проверка статуса сервиса (возвращает 200 OK и timestamp).
GET /api/metrics — базовая статистика обработанных запросов.

## AI-интеграция и надежность (Fallback)
При обработке формы комментарий передается в OpenAI для анализа тональности и генерации ответа.
Ключевым требованием была надежность: если API-ключ не указан, истек таймаут (5 секунд) или OpenAI вернул ошибку, сервис перехватывает исключение, логирует его и использует заранее подготовленный шаблонный ответ. Это гарантирует, что пользователь в любом случае получит успешный ответ от формы.

## Роль AI-ассистента в разработке кода
В процессе выполнения задания для ускорения работы использовался AI-ассистент.
- **Делегировано AI:** Генерация шаблонного кода и атрибутов OpenAPI для Swagger-документации, написание базовых конструкций `try-catch`.
- **Реализовано вручную:** Архитектура, бизнес-логика в `ContactService`, правила валидации и настройка окружения. AI-инструменты сэкономили время на рутине, но требовали постоянного ручного ревью и адаптации под специфику Laravel 11.

**Примеры использованных промптов:**
1. *«Напиши PHP 8 атрибуты OpenAPI (OA\Post, OA\RequestBody) для Laravel контроллера, который принимает JSON с полями name, email, phone, comment. Добавь описания для ответов 200, 422, 429, 500».*
2. *«Предложи структуру try-catch блока для Laravel HTTP-клиента при запросе к OpenAI API. Нужно логировать ошибки через Log::error и возвращать fallback-строку, если API недоступен или вернул не 200 OK».*

## Хранение данных
Поскольку использование базы данных в задании было опциональным, я выбрал прагматичный подход для хранения служебных данных, чтобы не усложнять архитектуру:

Логи: Записываются в стандартный файл storage/logs/laravel.log. Эмуляция отправки email также фиксируется в логах.

Rate Limiting: Используется встроенный middleware Laravel throttle, который в локальном окружении хранит счетчики в файловом кэше.

   Метрики: Эндпоинт /api/metrics анализирует лог-файл и подсчитывает количество успешных обработок. Для целей тестового задания это рабочее решение; в production-окружении для этих задач, разумеется, лучше использовать Redis или реляционную БД.
